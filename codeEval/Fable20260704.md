# Code Quality Evaluation — terrestrial-brain

**Date:** 2026-07-04
**Reviewer:** Claude Fable 5
**Method:** Four parallel in-depth review passes (MCP server core + tools; extractor pipeline + ingest; Obsidian plugin; tests + database + infrastructure), all source files read in full (~15,000 lines), findings cross-checked against the owner's standing standards (short single-purpose methods, no cryptic names, abstraction via swappable interfaces, mock-boundary rule, zero skips, GDPR-conscious design, one-command dev startup).

---

## Executive Summary

**Overall grade: C+ / B-.** This is a functional, honestly documented, and in places genuinely thoughtful codebase — the integration test suite is real (HTTP → edge function → Postgres, zero mocks on tested paths), the extractor pipeline has an actual pluggable interface, migrations are disciplined, and the README leads with its own known weaknesses. But it is built largely from **copy-paste rather than abstraction**: the same error-envelope, LLM-call, HTTP-route, and name-resolution boilerplate is duplicated dozens of times; there is **no data-access layer** anywhere (raw supabase-js calls inline in every handler); several **god functions** run 100–260 lines; and the Obsidian plugin is a single 802-line file. There are also a handful of **real correctness and security defects**: a `people` table RLS policy open to anyone with the anon key, LLM-driven hard deletes of user thoughts, non-atomic destructive update sequences, a vault-sync command that reports success when every note failed, an unverified public Slack ingest endpoint, and the API key transmitted in URL query strings on both client and server. None of these are exotic — they are exactly the gaps a strong review process (or CI, which does not exist) would have caught.

## Cross-Cutting Themes

These issues surfaced independently in multiple review passes — they are systemic, not local, and fixing them centrally yields the most leverage.

### X1. Copy-paste instead of abstraction (the defining weakness)
- **MCP error/response envelope** hand-built ~60 times; identical `catch` blocks in every handler (`tools/thoughts.ts:164-169`, `tools/tasks.ts:51-56`, `tools/projects.ts:46-51`, and ~25 more).
- **OpenRouter LLM call block** copy-pasted 6× across 4 extractor files plus `helpers.ts` and `thoughts.ts` — same base URL constant, same headers, same parse/validate/fallback shape (`date-parser.ts:260-325`, `people-extractor.ts:33-112`, `project-extractor.ts:67-126, 193-258`, `task-extractor.ts:238-301, 387-481`).
- **Batch ID→name resolution** reimplemented 8+ times with divergent semantics, while a helper for it already exists but is used by only one file (`helpers.ts:26-52` vs. inline copies in `tasks.ts`, `queries.ts`, `ai_output.ts`, `projects.ts`, `documents.ts`).
- **Six near-identical HTTP route blocks** in `index.ts:69-203`.
- **Test `callTool` helper** copy-pasted into 8 test files (13 helper copies total, plus repeated hardcoded URL/key constants).
- **Extractor list** `[new ProjectExtractor(), new PeopleExtractor(), new TaskExtractor()]` duplicated verbatim at 4 call sites where ordering is load-bearing (`tools/documents.ts:49, 324`, `tools/thoughts.ts:485, 889`).
- Plugin `callIngestNote` is a near-verbatim copy of `callHTTP` (`obsidian-plugin/src/main.ts:406-448`).

### X2. No swappable interfaces (direct violation of the owner's abstraction rule)
The only genuine interfaces in the repo are `FunctionCallLogger` (`logger.ts:39-55`) and the extractor pipeline's `Extractor` contract. Everything else is concretely coupled: no repository layer over supabase-js (~50 inline queries), no `AiProvider` over OpenRouter, no `TerrestrialBrainApiClient` in the plugin. Consequence: nothing can be unit-tested without a live DB / paid LLM API, and the test suite's LLM dependency (X6) is unfixable until this seam exists.

### X3. God functions (violation of the short-methods rule)
- `handleIngestNote` — 256 lines, ~7 responsibilities (`tools/thoughts.ts:844-1100`)
- `TaskExtractor.extract` — ~245 lines with five literal `--- Phase N ---` comments begging to be methods (`extractors/task-extractor.ts:490-734`)
- `get_project_summary` (~220 lines, `tools/queries.ts:39-261`), `get_recent_activity` (~220, `queries.ts:279-500`), `create_tasks_with_output` (~160, `tools/ai_output.ts:257-419`), `update_thought` with a fully duplicated branch (~145, `thoughts.ts:584-731`), plugin `onload` (~100, `main.ts:76-178`), modal `onOpen` (~95, `main.ts:531-625`), `reconcileCheckboxes` (~108, `task-extractor.ts:107-215`).

### X4. API key in URL query strings, end to end
Server accepts `?key=` (`index.ts:62`) with `cors origin: "*"` and a non-constant-time compare; the plugin puts the key in the endpoint URL (`main.ts:664-667, 771-779`). Keys land in edge/proxy logs, browser history, and Referer headers. Both sides need to move to a header (`x-brain-key` / `Authorization: Bearer`).

### X5. No fail-fast env handling
`Deno.env.get(...)!` at module level in 6+ files (`index.ts:24-26`, `helpers.ts:4`, `thoughts.ts:13`, all four LLM-calling extractor files). A missing `OPENROUTER_API_KEY` becomes `Authorization: Bearer undefined` at request time; a missing `MCP_ACCESS_KEY` silently breaks auth. This also forces the `(globalThis as any).Deno` stub hack in `project-extractor.test.ts:4-15`.

### X6. Tests depend on a live, paid, nondeterministic LLM API
Embeddings and extraction run through OpenRouter with no stub mode, so a green test run costs money and can flake on model behavior; nothing documents the `OPENROUTER_API_KEY` prerequisite. Several tests hedge around LLM unavailability with `if (!result.includes("No thoughts found"))` guards — **skips wearing a "passed" badge**, violating the zero-skip rule in spirit and failing the GATE 2b mutation check (`tests/integration/thoughts.test.ts:69-76, 146-148, 253-256, 343-354, 410-416`; `extractors.test.ts:1222-1232`).

### X7. GDPR / data-lifecycle gaps
- `function_call_logs` stores full tool inputs (personal note content) + IP addresses **forever** — no retention/purge policy, no size cap (`migrations/20260404000002`).
- The plugin syncs every note by default (opt-out tag) and offers **no deletion pathway**: deleting a vault note never deletes it from the backend (`main.ts:481-490` only forgets the local hash).
- LLM-driven **hard deletes** of thoughts contradict the system's own soft-archive convention (see M3).

### X8. No CI, no one-command startup
No `.github/` workflows at all; dev requires manually running `supabase start`, the plugin build, and `deno test` (whose command the README gets wrong — it says `npx vitest run` but the suite is Deno, `README.md:376`). Both violate the owner's standing rules ("one script starts everything"; "all test layers, every time" currently depends entirely on manual discipline).

### Security

## Medium/Low Findings by Area

### MCP server core + tools
- **No shared DB row types** — the `thoughts` row shape is hand-retyped inline in at least 4 places; `select("*")` with untyped access is common. Generate types (`supabase gen types typescript`) and use `SupabaseClient<Database>`. (`thoughts.ts:110-119, 243, 897`; `projects.ts:158-162`; etc.)
- **`withMcpLogging` erases handler typing** with `any[]` + 4 lint-ignore pragmas, and doesn't guard the handler with try/catch — making it generic and adding the catch would also let ~30 per-handler catch blocks be deleted. (`logger.ts:136-149`)
- **Loose zod schemas**: `status`/`type`/`reliability` are `z.string()` with valid values only in descriptions; most `*_id` params lack `.uuid()` while a few have it. Invalid values go straight to the DB.
- **Two naming generations coexist** — older files (`thoughts.ts`, `tasks.ts`, `projects.ts`, `helpers.ts`) are full of `q`, `t`, `m`, `r`, `d`, `p`, `c`, `k`, `v`, `o`, `qEmb`, `kids` (direct violation of the no-single-letter rule); newer files (`people.ts`, `documents.ts`, `queries.ts`) are correctly named. Mechanical rename pass needed.
- **Inconsistent conventions**: not-found is `isError: false` in `get_thought_by_id`/`get_document` but `isError: true` in `get_project`/`get_person`; zero-field updates are an error in some tools, a friendly no-op in others; `update_task`/`update_project` report success for nonexistent UUIDs (no affected-row check).
- **Unbounded queries**: `list_*` limits have no `.max()`; `thought_stats` loads *every* thought into memory to count client-side (`thoughts.ts:309-315`) — should be a SQL RPC.
- **`ilike` pattern injection** (user `%`/`_` act as wildcards; `%` matches everything) and unindexable full-content `ilike` scans (`documents.ts:196-197`).
- Cruft: server self-identifies as `"open-brain"` (`index.ts:34`); stale "top 10" comment vs. `slice(25)` (`queries.ts:102-112`); magic numbers 25/20/10/80; usefulness reminder emitted as both header *and* footer of every search response (`thoughts.ts:155-163, 268-277`) — intentional prompt-engineering or bloat, uncommented either way.

### Extractor pipeline + ingest
- **Marker vocabulary defined in 3 places and already drifting** (`due|by|deadline|before` in `date-parser.ts:146` re-hardcoded twice in `task-extractor.ts:96-105`; assignment markers duplicated at `:322`).
- **Extractors perform DB writes inside "extraction"** — the `ExtractionResult` contract implies pure detection but the real contract is detect + mutate + enrich; mid-pipeline failure leaves partial writes. Document the side-effect contract at minimum; ideally return intents the runner applies.
- **O(checkboxes × tasks × len²) character-level LCS**, run twice, inside a CPU-limited edge function — needs cheap prefilters (exact match, length ratio, token Jaccard) before DP.
- **Parser gaps**: parent detection requires exact `depth - 1` (a 0→2 jump orphans the subtask) and scans backward across section boundaries; checkbox regex accepts only `- [ ]` (not `* [ ]`/`+ [ ]`); 2-space indent assumption where Obsidian defaults to 4. Bare ISO date pattern matches inside URLs/version strings and strips them from task text (`date-parser.ts:169-176`). `\W` boundary checks are not Unicode-aware ("José"). "next Monday" treated identically to "Monday".
- **Slack dedup is content-based** ("same text within 5 min") — drops legitimate repeats and ignores `event_id`/`event.ts`, which exist precisely for retry idempotency (`ingest-thought/index.ts:165-179`).
- **Dead code**: `containsDateLikeWords` + `inferDatesFromContent` (~80 lines incl. a whole LLM prompt path, `date-parser.ts:246-325`) and `logFunctionError` (`ingest-thought/index.ts:39-51`) are called nowhere. The "fire-and-forget logging" banner is inaccurate (it's awaited on the critical path).
- **Zero unit tests for the most complex files** — `parser.ts`, `date-parser.ts`, `task-extractor.ts`, `people-extractor.ts`, `pipeline.ts` have none, despite exporting a dozen pure helpers clearly shaped for testing (and `referenceDate` injection existing specifically for it). The similarity/containment thresholds (0.8/0.85) and date year-inference are exactly the surfaces that regress silently.
- Good news: `ingest-thought` calls the MCP server's `capture_thought` over JSON-RPC rather than reimplementing extraction — a single code path (only nit: positional `result.id === 2` coupling).

### Obsidian plugin
- **Monolith**: plugin class, sync engine, debouncer, HTTP client, poller, modal, settings tab, and utilities all in one file; should be ~6 modules with `main.ts` as a thin composition root.
- **Business logic entangled with Obsidian API** — only the 5 exported utilities are testable without mocks; tests must build fake plugins via `Object.create(prototype)` and overwrite private members (`main.test.ts:68-129`). `generateCopyPath` with its injected `existsCheck` (`main.ts:781-802`) is the pattern the rest should follow.
- **Mock-only test coverage**: the plugin's real `callHTTP`/`callIngestNote`/`buildEndpointUrl` code is never run against the real backend anywhere in the repo (the integration tests use their own hand-rolled fetch helpers) — a server response-shape change breaks the plugin while every test stays green. Some tests assert mock plumbing (`addRibbonIcon` called with args; `saveSettings` mocked so bug C10 was invisible).
- **Server responses cast, never validated** (`as AIOutputMetadata[]`, `main.ts:234, 282`); `strict` mode off (only `strictNullChecks`), so implicit `any` flows through settings migration.
- **Dead setting**: `projectsFolderBase` is declared, rendered in settings UI, persisted — and never read anywhere (verified by grep). Users can configure a knob that does nothing.
- **Config drift**: manifest 1.1.0 vs package.json 0.3.0, no `versions.json`; esbuild 0.17 (Jan 2023), TS 5.3, `@types/node` 18 (EOL), `"obsidian": "latest"` unpinned; 1,400-char manifest description; ~25 inline `element.style.*` assignments instead of a `styles.css`; misplaced/orphaned JSDoc (`main.ts:762-766`); magic `60000` ×4 and `2000`; `if (!content && content !== "")` for `content === undefined`.
- No retry/backoff/offline strategy; failed manual "Pull AI Output" is silent (`console.error` only); error Notices echo raw server response bodies.

### Tests
- **Order-dependent tests with shared module state** (`projects.test.ts:39-50`, `tasks.test.ts:36`, `thoughts.test.ts:738-741`) — an early failure cascades, and tests can't run individually with `--filter`. `projects`/`tasks` tests never clean up their fixtures (archived rows accumulate across runs).
- **Cleanup implemented as vacuous test cases** (`assertEquals(true, true)` — `extractors.test.ts:1948-1959` and ~9 more) inflating the pass count; rows orphan if a test fails before registering its ID. Use `try/finally`.
- **Unit tests live in `tests/integration/`** (`parse.test.ts` entirely; ~half of `extractors.test.ts`), plus a stray in-source test file (`project-extractor.test.ts`, whose Deno-stub construction is fragile and half of which just verifies its own fetch mock). Split `tests/unit/` from `tests/integration/`.
- Assertions verify real DB side-effects (good) but often match human-facing prose/emoji — copy tweaks break tests. Acceptable for an MCP text protocol; noted as coupling.
- No `deno.json` test task; README documents the wrong runner entirely (`npx vitest run` → should be `deno test --allow-net --allow-env tests/integration/`).

### Database
- **`match_thoughts` pasted in full 4 times** across migrations (drop-with-exact-prior-signature + full re-paste per change) — correct but the source of truth is "whichever migration sorts last." Adopt a canonical always-latest file or declarative schema for functions.
- **Nullability/naming drift**: early tables allow `created_at NULL`, later ones don't; index naming flips convention (`thoughts_created_at_idx` vs `idx_people_type`); `documents."references"` reserved-word column; thought→project links live in JSONB with **two coexisting formats** (`project_id` string vs `projects` array) that every consumer must handle, no FK integrity (orphaned UUIDs are expected behavior, with a test blessing it).
- `function_call_logs` lacks a `(function_name, called_at)` index and any check constraints, on top of the retention issue (X7).
- Idempotency is adequate for Supabase's linear model; the `ai_notes → ai_output` data migration was well executed.

### Scripts / repo hygiene
- `PROJECT_REF` scraped from CLI output by matching the `●` glyph with awk — brittle; prefer `supabase/.temp/project-ref` or `--output json`. Stale filename in usage comments (`deploy-prod.sh`).
- SELinux `setenforce` workaround is one developer's machine policy baked into a shared script — gate behind an env flag.
- Stock Node `.gitignore` (Bower/Gatsby/VuePress noise); orphan `tests/node_modules/` with no `tests/package.json`; tracked ad-hoc `Planning/` directory duplicating openspec's role; no lint/format config beyond `.vscode` (tests uncovered by any linter).
- Docs are otherwise strong and current (README's 31 tools verified accurate; upgrade/fresh-install guides are genuinely good); openspec is actively maintained (~30 archived changes, clean in-flight state), with minor duplicate dir-and-file spec entries.

## What Was Done Well

Fairness requires noting this codebase's real strengths:

1. **The integration tests are real integration tests** — HTTP → edge function → Postgres with effects verified by independent direct DB queries, zero mocks on tested paths, ~319 cases including auth-denial, error paths, empty-array edges, and re-ingest idempotency. Fully honors the mock-boundary rule.
2. **The extractor pipeline has genuine architecture**: a real `Extractor`/`ExtractionContext` contract, layered cheap-deterministic-before-LLM heuristics, and LLM outputs validated against ID allowlists so the model can't inject foreign IDs — a defensive pattern applied consistently.
3. **`FunctionCallLogger` is a proper interface** with a decorator applying it cross-cuttingly, and logging failures never break tools.
4. **No N+1 queries** — name resolution is consistently batched; parallel `Promise.all` where it matters.
5. **Thoughtful product-level design**: two-phase AI-output fetch (metadata first, content only after user acceptance), hash-based dedup with cross-feature awareness (delivered output pre-registers its hash so the write doesn't re-ingest), settings migration with legacy cleanup, `pollInProgress` guard.
6. **Migration discipline**: linear, never-edited, well-commented history; correct partial index for the pending-output poll; appropriate HNSW/GIN/btree choices.
7. **Honest documentation** — the README discloses its own known security weakness up front; MCP tool descriptions are unusually well-written; `validators.ts` is exemplary (single-purpose, well-named, edge-cases handled, actually unit-tested).
8. **Naming in the newer code is excellent** — the extractors follow the no-single-letter rule everywhere, including LCS inner loops (`candidateA`, `previousRow`, `checkboxIndex`).

The pattern is visible: **later code is markedly better than earlier code** (naming, error handling, RLS policies, test rigor all improved over time). The main debt is that the earlier code was never brought up to the later standard, and cross-cutting seams (repositories, LLM client, shared envelope) were never extracted.
