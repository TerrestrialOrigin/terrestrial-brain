## Why

Steps 10–12 provision, operate, and end a customer's brain, but every one of them persists its state — including the minted MCP access key and the project db password — in **interim 0600 plaintext files** that are explicitly documented as "an accepted interim until Step 13's control plane." There is no durable place that maps a customer to their project, tokens, and subscription status; no proper secret storage; no home for the export artifacts Step 12 delivers; and no backend the future dashboard (Steps 16/17) and Paddle billing webhooks (Step 14) can build on. Step 13 introduces that missing spine: **one ordinary Supabase project** as the durable control plane, replacing the file stores with database-backed stores whose atomic claim is enforced by Postgres, providing real secret storage, tracking subscription status and export retention, and exposing a transport-neutral backend service the commercial layers consume.

## What Changes

- **New hosted-side capability, in the SAME SEPARATE PRIVATE repo as Steps 10–12** (`terrestrial-brain-hosting`, branch `feature/ControlPlane`). The control plane holds the mapping of customers → provisioned projects → MCP tokens → subscription status, plus the privileged secrets (access keys, db passwords) — hosted-business logic that must never ship in the FSL-1.1-MIT public tree. This public repo keeps only the OpenSpec artifacts, one `ThreatModel.md` entry (T28), and the plan checkbox — **no control-plane code**.
- **A control-plane data model** — the schema of "one ordinary Supabase project" (a normal project, not a per-customer one): `customers`, `customer_projects` (the durable provisioning record: project ref, region, mcp_url, subscription status, plus the resume cursor), a secret store for the minted access key + db password, `deprovision_jobs`, and `export_artifacts` (the retention record Step 12 deferred). Every table is service-role-only (the control plane is operator/backend-only; there is no customer-facing RLS surface yet).
- **Durable, DB-backed job stores replacing the interim file stores** — new implementations of the **existing** `ProvisioningJobStore` and `DeprovisionJobStore` ports, backed by the control-plane project, whose atomic claim/compare-and-swap is enforced by a **Postgres conditional write** (not a process-local file check). **No change to any step or pipeline logic** — the stores are seams; provisioning/fleet/deprovision code is unchanged. The interim file stores are retained as an explicit dev fallback selected by config, never in the hosted path.
- **Proper secret storage** — the minted MCP access key and the project db password move out of world-persisted plaintext files into a service-role-only secret store in the control-plane project (**BREAKING** for the interim file layout: hosted deployments read/write secrets via the control plane, not `TB_JOBSTORE_DIR`). The secret store is a seam so a stronger backend (Vault / envelope encryption) can be swapped in later without touching callers.
- **Export-artifact retention/rotation** — the open item Step 12 deferred: `export_artifacts` records each delivered dump (location + checksum + byte/row counts + delivered-at + retention-until — never content), and a new `purge-exports` operation deletes artifacts past their retention window (idempotent, counted outcome, bounded query).
- **A transport-neutral control-plane service layer** — a thin, typed service (list/get customers, read a customer's connection + subscription status, set subscription status, record/list export artifacts) that the future dashboard (Steps 16/17) and Paddle billing webhooks (Step 14) call. This is the **backend/data layer only** — **NOT** the Paddle integration (Step 14) and **NOT** the dashboard UI (Steps 16/17).
- **New narrow seam, reusing Steps 10–12's composition root:** a `ControlPlaneClient` port (the narrow set of reads/writes the DB-backed stores + service layer need) with a deterministic in-memory fake that faithfully models the Postgres conditional-write claim semantics, so the whole control plane — claim/CAS, idempotency, crash-resume, retention purge — runs in tests with no live Supabase.
- **CLI + tasks:** a `control-plane` maintenance command (apply the control-plane schema, `purge-exports`), plus an opt-in fail-loud live smoke against a throwaway control-plane project.
- **A ThreatModel entry (T28)** for the control-plane surface: the control plane now concentrates every customer's secrets and mapping in one project (a high-value target), so the analysis covers secret-at-rest storage, service-role-only access, DB-enforced atomic claim, and export-artifact retention.

## Non-goals

- **The Paddle billing integration itself** (Step 14) — Step 13 provides the subscription-status column and the service methods to read/write it and the webhook-idempotency-friendly data model, but does not verify signatures, handle webhook payloads, or drive checkout.
- **The dashboard / onboarding UI** (Steps 16/17) — Step 13 is the backend/data layer those screens call; it ships no UI.
- **Customer-facing auth / RLS on the control plane** — the control plane is operator/backend-only in this step (service-role access); a customer-scoped API surface is future work.
- **Managed-AI usage metering** (Step 15) — out of scope here.
- **Rewriting provisioning/fleet/deprovision logic** — those steps are unchanged; only the store *implementation* behind their existing seams changes.
- **A stronger secret backend (Vault / envelope encryption / KMS)** — the secret store is a seam; this step lands a service-role-only DB store with the trade-off documented, leaving a stronger backend as a future swap.
