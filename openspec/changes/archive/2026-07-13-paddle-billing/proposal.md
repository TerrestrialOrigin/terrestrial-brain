## Why

Step 13 landed the durable control plane (customers → projects → tokens → subscription status) and left the subscription-status write (`ControlPlaneService.setSubscriptionStatus`) sitting unused — its own comment names "the Paddle billing webhooks (Step 14)" as the caller that would drive it. There is still no way for a real payment event to turn into a provisioned brain, a paused project, or a deprovisioned tenancy: no webhook receiver, no signature verification, no idempotent at-least-once handling, and no mapping from Paddle's lifecycle events onto Steps 10/12's pipelines. Step 14 introduces that missing commercial spine — a verified Paddle webhook that drives **checkout → provision**, **payment failure → pause**, and **cancellation → deprovision** — so that a paying customer's lifecycle actually operates the fleet.

## What Changes

- **New hosted-side capability, in the SAME SEPARATE PRIVATE repo as Steps 10–13** (`terrestrial-brain-hosting`, branch `feature/PaddleBilling`). The billing webhook maps a customer's payment lifecycle onto their provisioned project + subscription status — hosted-business logic that must never ship in the FSL-1.1-MIT public tree. This public repo keeps only the OpenSpec artifacts, one `ThreatModel.md` entry (T29), and the plan checkbox — **no billing code**.
- **Webhook signature verification** behind a new `PaddleSignatureVerifier` seam: HMAC-SHA256 over the `ts:rawBody` payload extracted from the `Paddle-Signature` header (`ts=…;h1=…`), constant-time compared against the configured webhook secret, with a bounded timestamp-freshness window so a captured request can't be replayed indefinitely. An invalid or stale signature is rejected before any parsing or side effect.
- **Parse-at-boundary Paddle event parsing** into a known-good discriminated union — the events we act on (`transaction.completed`/`subscription.activated` → provision, `transaction.payment_failed`/`subscription.past_due` → pause, `subscription.canceled` → deprovision, `subscription.resumed` → resume). The internal customer id and region are read from Paddle `custom_data`; an event with no resolvable customer is refused, never invented. Unrecognised event types are acknowledged and ignored (a `200`, no action) rather than erroring.
- **At-least-once idempotent handling** via a new `webhook_events` table in the control plane keyed on the Paddle event id, claimed with a single atomic insert-if-absent conditional write: a first delivery claims and processes; every redelivery of the same event id is a no-op that still acknowledges. This makes the non-idempotent side effects (pause, resume) safe under Paddle's at-least-once redelivery, complementing the already-idempotent subscription upsert and provisioning/deprovision claims.
- **A transport-neutral `BillingService`** (the Rule-of-Three "one logic, thin adapters" seam Step 13 anticipated): `handleWebhook(rawBody, signatureHeader)` → verify → parse → claim-event → dispatch, returning a transport-neutral result the HTTP layer maps to a status code. Event dispatch maps each lifecycle event onto the existing pipelines through narrow trigger seams (`ProvisioningTrigger`, `DeprovisionTrigger`) and the control-plane service, so the mapping is written once and unit-tested with fakes.
- **Project pause/resume** added to the existing `SupabaseManagementApi` seam (`pauseProject`/`restoreProject`) so a payment failure pauses the running project (plan: "Payment failure → pause project") and a recovered payment restores it; the subscription status moves to `paused`/`active` in lock-step.
- **The webhook HTTP receiver** as a thin `billing serve` command (`Deno.serve`) that only adapts HTTP ↔ `BillingService.handleWebhook` — reading the raw body + `Paddle-Signature` header and returning the mapped status. All logic lives in the transport-neutral service; the HTTP layer holds none.
- **Config + CLI + tasks:** new `TB_PADDLE_WEBHOOK_SECRET` (crown-jewel, redacted, required only for the `billing` command) and `TB_PADDLE_DEFAULT_REGION`; a `billing serve` command; an opt-in, fail-loud `billing:live` smoke.
- **A ThreatModel entry (T29)** for the webhook surface: an internet-facing, unauthenticated-by-default HTTP endpoint whose events trigger money-relevant and destructive actions (provision → cost, deprovision → data deletion), so the analysis covers signature forgery, replay, event/customer spoofing, and the secret handling.
