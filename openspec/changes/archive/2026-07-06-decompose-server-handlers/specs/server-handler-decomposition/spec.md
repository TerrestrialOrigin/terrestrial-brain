## ADDED Requirements

### Requirement: Composite-query tools SHALL separate data-gathering from formatting

The `get_project_summary` and `get_recent_activity` tools SHALL be structured as a data-gathering function that performs all I/O and returns a plain data structure, plus a pure formatting function that takes that structure and returns the response text. The formatting function SHALL perform no I/O (no DB, no network) so it is deterministically unit-testable with synthetic data. The observable output text of each tool SHALL remain identical to the pre-refactor output for the same inputs and data.

#### Scenario: Formatter renders a populated project summary deterministically

- **WHEN** the project-summary formatter is called with a synthetic data structure containing a project, children, open tasks, and recent thoughts
- **THEN** it returns the same markdown (headings, counts, task/thought lines, usefulness reminder) the tool produced before the refactor, without touching the database

#### Scenario: Formatter distinguishes a failed sub-query from an empty result

- **WHEN** the formatter is given a section whose query result carries an `error`
- **THEN** it renders the explicit `(section unavailable: <reason>)` marker for that section rather than the empty-state prose, and renders the empty-state prose only for a successful zero-row result

#### Scenario: Recent-activity dedup is a single shared helper

- **WHEN** recent activity contains an entity appearing in both the "created" and "updated" result sets
- **THEN** the entity is de-duplicated by name via one shared `dedupeByName` helper used for both projects and people, and the rendered activity lists the entity exactly once with its "created" action taking precedence

### Requirement: The note-ingestion handler SHALL be composed of named single-purpose steps

`handleIngestNote` SHALL delegate to named step functions — checking whether content is unchanged, upserting the note snapshot, fetching existing thoughts, requesting the reconciliation plan, executing the reconciliation plan, and formatting the summary — with the top-level handler acting as a thin orchestrator. Each step SHALL be independently unit-testable. The end-to-end behavior (unchanged-skip, fresh-ingest fallback, reconciliation updates/adds/soft-archives, and the summary message) SHALL remain identical.

#### Scenario: Unchanged content is skipped

- **WHEN** a note is ingested whose stored snapshot content equals the incoming content
- **THEN** the handler returns the "Note unchanged — skipped." success result without running the reconciliation plan

#### Scenario: Reconciliation delete list soft-archives rather than hard-deletes

- **WHEN** the reconciliation plan's `delete` list names an existing thought id
- **THEN** executing the plan archives that thought (sets `archived_at`) and never issues a hard delete, so a hallucinated id cannot destroy captured knowledge

#### Scenario: Unparseable reconciliation plan falls back to a fresh ingest

- **WHEN** the reconciliation-plan request yields output that cannot be parsed into the plan shape
- **THEN** the handler falls back to a fresh ingest of the note (identical to pre-refactor behavior), while a transport-level failure instead propagates as an error

### Requirement: Usefulness reminders SHALL be produced by one tone-parameterized builder

The reminder text appended by thought-returning tools SHALL be generated by a single builder parameterized by a tone (hard / soft / terse) rather than by duplicated per-tool string literals. The exact reminder text for each tone SHALL match the pre-refactor output.

#### Scenario: Each tone produces its established text

- **WHEN** the builder is invoked with a set of candidate thought ids for the hard, soft, and terse tones respectively
- **THEN** it returns the required-action reminder, the softer browsing reminder, and the terse one-line reminder respectively, each embedding the candidate ids exactly as the corresponding tool did before the refactor

### Requirement: The update-thought handler SHALL not duplicate its field-application logic

`update_thought` SHALL apply reliability, author, and project/document reference updates through a single code path shared by the content and non-content cases, with the content case additionally regenerating the embedding and metadata. Validation of "at least one field provided", the replace-semantics for references, preservation of `metadata.source`, and the emitted confirmation text SHALL remain identical.

#### Scenario: Non-content update omits AI regeneration and metadata when references unchanged

- **WHEN** `update_thought` is called with only `reliability` (no content, no reference changes)
- **THEN** no embedding or metadata regeneration occurs, the update payload carries no `metadata` key, and the confirmation lists the updated field exactly as before

#### Scenario: Content update regenerates embedding and metadata

- **WHEN** `update_thought` is called with new `content`
- **THEN** the embedding and metadata are regenerated and merged (preserving `metadata.source` and applying any reference overrides), and the confirmation is prefixed with the "content (embedding + metadata regenerated)" label

### Requirement: create_tasks_with_output SHALL split validation, insertion, and formatting while preserving atomicity

`create_tasks_with_output` SHALL be decomposed into distinct validation, name-resolution, atomic-insertion, and markdown-formatting steps. The up-front `parent_index` validation and the mid-loop rollback of already-inserted task rows on any insert failure SHALL be preserved exactly.

#### Scenario: A mid-loop insert failure rolls back already-inserted tasks

- **WHEN** inserting the Nth task fails after tasks 1..N-1 were inserted
- **THEN** the already-inserted task rows are deleted and an error is returned, leaving no orphaned rows

#### Scenario: A forward parent_index is rejected before any insert

- **WHEN** a task references a `parent_index` that points to a later task
- **THEN** the handler returns a validation error and creates zero task rows

### Requirement: No tool handler SHALL exceed the single-purpose size checkpoint

After the refactor, no function in `supabase/functions/terrestrial-brain-mcp/tools/*.ts` SHALL substantially exceed the ~50-line single-purpose checkpoint, and the existing integration test suite SHALL remain green without any test edits.

#### Scenario: Integration suite passes unchanged

- **WHEN** the full Deno integration suite is run against the refactored handlers
- **THEN** it passes with zero failures and zero edits to the integration test files
