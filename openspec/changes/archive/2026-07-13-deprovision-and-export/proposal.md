## Why

Step 10 provisions a customer brain and Step 11 operates the fleet, but nothing **ends** a customer's tenancy. When a customer cancels (or exercises their GDPR rights), two things must happen and must happen in the right order: they must receive a complete, verified export of their data (portability, GDPR Art. 20), and then their project must be permanently deleted (erasure, GDPR Art. 17). Doing this by hand risks the worst failure mode in the whole system — deleting a customer's project before their data has actually been handed back. Step 12 automates cancel → full verified data dump delivered → project deleted, as one idempotent, resumable, honest operation, and doubles as the whole-account GDPR export/erasure pathway (complementing fix-plan Step 25's per-note `forget_note`).

## What Changes

- **New hosted-side capability, in the SAME SEPARATE PRIVATE repo as Steps 10 & 11** (`terrestrial-brain-hosting`, branch `feature/DeprovisionAndExport`). Deprovisioning wields the privileged Management token to *delete* customer projects and reads *all* customer content into an export — hosted-business logic that must never ship in the FSL-1.1-MIT public tree. This public repo keeps only the OpenSpec artifacts, one `ThreatModel.md` entry (T27), and the plan checkbox — no deprovision/export code.
- **A non-destructive `export` operation** — produce a complete logical dump of a customer's brain (all tables), deliver it to the customer's configured destination, and **verify** the delivered artifact (size + checksum) before reporting success. Repeatable and side-effect-free (satisfies an active customer's GDPR data-portability request without ending their tenancy).
- **A destructive `deprovision` operation** — export + verify + **then** delete the project, as a step-cursor state machine (`EXPORT_DATA → VERIFY_EXPORT → DELETE_PROJECT → FINALIZE`) that is idempotent and resumable. **BREAKING for the target customer only** (their project is permanently deleted); it never touches any other customer.
- **The delete is gated on a verified export** — the destructive `DELETE_PROJECT` step is unreachable until `VERIFY_EXPORT` confirms the dump was delivered intact. A crash after export leaves a recoverable state (re-run resumes at delete); a crash before a verified export never deletes anything. This is the runs-twice / crashes-halfway / interleaves invariant applied to the one genuinely irreversible operation in the product.
- **Honest counted outcome** — a customer is reported `deprovisioned` only when the export verified AND the delete succeeded; a failure at any step leaves a recoverable `failed` job with its error, never a green "done."
- **Fleet consistency** — on completion the customer's provisioning job flips to a new `deprovisioned` status so Step 11's fleet enumeration (which counts only `done` jobs) no longer operates on the deleted project.
- **New narrow seam, reusing Steps 10/11's composition root:** a `DataExporter` port (`export` produces + delivers a dump and returns a manifest; `verify` re-reads the delivered artifact to confirm integrity) with a deterministic fake, plus a dedicated `DeprovisionJobStore` for the deprovision job's atomic claim + resumable cursor. The existing `SupabaseManagementApi.deleteProject` and `ProvisioningJobStore` (read the customer's `projectRef`/`dbPassword`, flip status) are reused.
- **CLI additions:** `export` (non-destructive dump + verify) and `deprovision` (export + verify + delete), plus an opt-in fail-loud `export:live` smoke.
- **A ThreatModel entry (T27)** for the deprovision/export surface: content-bearing export handling, destructive delete with the privileged token, the verify-before-delete safety gate, honest completion, and the GDPR retention consideration for the delivered artifact.

- **External APIs consumed:** Supabase Management API — reuses `deleteProject` (Step 10) for the destructive step and the read-only database-query / dump path for the export; the customer's project db password (persisted by Step 10) is used to produce the logical dump.
- **Secrets handled:** the same privileged Supabase Management API token (env-only, `Authorization: Bearer`, never in a URL, never logged) and the per-customer db password (used to run the dump, passed to the dump subprocess via the environment, never in argv/logs). Deprovision mints no new secret.
- **New personal-data flow:** unlike fleet ops' content-free reads, the export pulls full customer content into a delivered artifact — handled as a controlled, access-restricted delivery whose manifest records only location + checksum + byte/row counts (never content), with a documented retention posture (see ThreatModel T27).
- **Reuses, does not fork:** reuses the published-repo schema only insofar as the dump reflects it; copies nothing from OB1.
