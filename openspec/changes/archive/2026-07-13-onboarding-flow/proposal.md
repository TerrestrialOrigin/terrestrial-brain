## Why

Steps 10–15 built the whole hosted spine — provision a project (10), operate the fleet (11), deprovision + export (12), the durable control plane (13), the Paddle billing webhook that turns a payment into a provisioned brain (14), and per-tenant AI metering (15) — but there is no way for an actual customer to *walk in the front door*. Nothing takes a person from "I want this" to "here is my working brain": no signup, no place that opens Paddle checkout with the right `custom_data` for the Step-14 webhook to act on, no "your brain is being built…" wait, and — critically — no authorized surface that reveals the per-customer MCP access key so they can wire up Claude and Obsidian. The control plane deliberately never returns secrets through a customer-facing read (`ControlPlaneService` comment: "NEVER returns secrets"), so the one thing a new customer most needs (their token) is currently unreachable. Step 16 is that missing front door: the onboarding flow that composes Steps 10/13/14 into a signup → payment → provisioning-wait → connection-details experience.

## What Changes

- **New hosted-side capability, in the SAME SEPARATE PRIVATE repo as Steps 10–14** (`terrestrial-brain-hosting`, branch `feature/OnboardingFlow`). Onboarding reads the control plane, drives Paddle checkout, and reveals a per-customer access key — hosted-business logic that must never ship in the FSL-1.1-MIT public tree. This public repo keeps only the OpenSpec artifacts, one `ThreatModel.md` entry (T31), and the plan checkbox — **no onboarding code**.
- **A transport-neutral `OnboardingService`** (the Rule-of-Three "one logic, thin adapters" seam Step 13 anticipated for "Steps 16/17") with three operations: `startSignup({ email, region })` → creates the customer (idempotent `ensureCustomer`, status `none`), mints an opaque onboarding **session token** (bound to that one customer, stored **hashed** at rest), and returns the Paddle checkout `custom_data` (`customerId`, `region`, `email`) so the Step-14 webhook provisions the right customer; `getStatus(sessionToken)` → the "building…" poll (awaiting-payment / building / ready / failed / paused / canceled); `getConnection(sessionToken)` → the connection package, gated on provisioning `done`.
- **Provisioning is NOT triggered by signup.** Signup only creates the customer + session and hands back checkout parameters. The single provisioning trigger stays the Step-14 Paddle `transaction.completed` webhook — so there is exactly one provisioning entry point and no double-provision.
- **Session-token auth as the seam for "the authenticated customer."** The token is 256-bit CSPRNG, opaque, stored only as its SHA-256 hash (never in plaintext at rest), bound to a single customer id, bounded TTL, resolved by hash lookup, and carried in a request **header** (`x-tb-onboarding-token`) — never a cookie (no CSRF surface) and never a URL (no key-in-URL leak). Only the holder of a customer's session token can read that customer's status or connection package.
- **The connection package** — assembled only when provisioning is `done` — is `{ mcpUrl, accessKey, obsidianToken, header: "x-tb-key", claudeDesktopConfig, claudeCodeConfig }`. The access key is read from the service-role-only secret store (`ControlPlaneClient.getSecret`); the db password is never read or returned. Requesting the package before `done` is a distinct "not ready" state (HTTP 409), never a fake/empty package.
- **A thin `Deno.serve` HTTP receiver + three JSON endpoints** (`POST /onboarding/signup`, `GET /onboarding/status`, `GET /onboarding/connection`) plus two minimal, self-contained, accessible **served screens** (the "building…" poll screen and the "ready — here's how to connect" pitch screen). All logic lives in the transport-neutral service; the HTTP layer only maps outcomes to status codes and renders — mirroring Step 14's `billing/receiver.ts`.
- **Two new control-plane tables** behind the existing narrow `ControlPlaneClient` seam (no generic SQL): `onboarding_sessions` (token hash → customer id, created/expiry, service-role only). New client methods `createOnboardingSession` / `resolveOnboardingSession`; modelled in `InMemoryControlPlaneClient`.
- **A narrow `OnboardingTokens` seam** (`mintCustomerId()`, `mintSessionToken()`) — CSPRNG in the real adapter, deterministic fake in tests — kept separate from the MCP `KeyGenerator` so each generator stays single-purpose.
- **Config + CLI + tasks:** new `TB_PADDLE_CLIENT_TOKEN` + `TB_PADDLE_PRICE_ID` (public, non-secret — the browser opens Paddle.js checkout with them), `TB_ONBOARDING_SESSION_TTL_DAYS`, and `TB_PUBLIC_BASE_URL`; an `onboarding serve` command; an opt-in, fail-loud `onboarding:live` smoke.
- **A ThreatModel entry (T31)** for the onboarding surface: an internet-facing signup + a session-token-gated read that reveals a crown-jewel access key — covering token theft/guessing, customer enumeration, access-key exposure, CSRF, and signup abuse.

The db password is never read on this path.

- **Reuses, does not fork:** reuses Steps 10–14's composition root, config, redacting logger, `ControlPlaneService`, `ControlPlaneClient` secret store, and `mcpUrlFor`; the served HTML mirrors `billing/receiver.ts`'s thin-adapter shape; copies nothing from OB1.
